Architecture of a Resilient, AI-Assisted Form Automation Engine for Manifest V31. Introduction and Architectural ContextThe automation of data entry within enterprise web applications, government portals, and complex administrative systems presents a significant engineering challenge. Portals such as the Sri Lanka Inland Revenue Department (IRD) Revenue Administration Management Information System (RAMIS) are typically constructed using robust web frameworks such as ASP.NET, React, and Angular. These modern web architectures frequently generate highly volatile Document Object Models (DOMs). Elements are rendered asynchronously, identifiers (IDs) are auto-generated or obfuscated, and CSS class names are dynamically hashed by bundlers. Consequently, traditional browser automation strategies that rely on fixed DOM selectors or strict XPath queries suffer from extreme brittleness, breaking immediately upon minor front-end updates or layout shifts.To overcome these inherent limitations, this report details the architectural design of a highly resilient, production-grade form autofill engine. Operating entirely within the constraints of Google Chrome's Manifest V3 (MV3) extension architecture, this system executes completely in the browser environment without reliance on server-side automation tools, headless browsers like Puppeteer, or external backend infrastructure. Instead, the engine achieves human-like resilience by implementing semantic field detection, on-device Natural Language Processing (NLP), adaptive DOM abstraction, and mutation-aware state management. By parsing the DOM through the lens of web accessibility trees and visual heuristics, the engine dynamically interprets the intent of form fields, ensuring reliable execution even when the underlying markup undergoes substantial structural revisions.2. Manifest V3 Execution Environment and OrchestrationThe transition to Manifest V3 represents a paradigm shift in browser extension architecture, primarily designed to enhance user privacy, system security, and overall browser performance. For automation engineers, the most consequential modification in MV3 is the deprecation of persistent background pages in favor of ephemeral Service Workers. This necessitates a fundamentally distributed, state-independent architectural approach to form automation.2.1 The Ephemeral Service Worker ParadigmUnder previous manifest versions, automation engines could maintain global JavaScript objects in memory indefinitely, tracking the progress of a multi-step form over hours of user inactivity. In MV3, the background Service Worker is aggressively terminated by the browser. It shuts down after thirty seconds of event inactivity or a maximum of five minutes of continuous execution. Therefore, the central automation coordinator cannot rely on persistent local variables.To mitigate this, the architecture dictates that the Service Worker functions solely as an event-driven orchestrator and message router. All active automation states, schema mappings, and execution indices must be continually serialized and hydrated using asynchronous storage mechanisms. The Service Worker listens for explicit triggers, such as keyboard shortcuts or user interactions with the extension's user interface, and dispatches highly targeted command payloads to the specific Content Scripts operating within the active browser tab.2.2 Context Isolation and State PersistenceContent Scripts operate within an "isolated world," an execution environment that grants full access to the host page's DOM but completely isolates the extension's JavaScript variables from those of the host web application. This isolation prevents conflicts with global variables defined by the target site's Angular or React frameworks, but it also means the extension cannot directly read the host application's internal data models.To maintain continuity across complex, multi-page workflows, the engine leverages the suite of Chrome storage APIs. The chrome.storage.local API provides asynchronous, non-volatile storage suitable for retaining overarching configuration schemas and user profiles. However, reading and writing large datasets to disk-backed local storage can introduce latency bottlenecks during high-speed DOM mutation events. To address this, the architecture integrates the MV3 chrome.storage.session API. This API maintains data entirely in volatile memory, offering significantly lower latency for rapid read/write operations. It is restricted to a maximum capacity of one megabyte, making it the optimal repository for tracking the instantaneous progress of a multi-step form execution plan, recording which fields have been successfully mapped, and holding transient confidence scores generated by the semantic matching engine.3. Hardcoding document.getElementById('tax_field_2') introduces critical failure points. Instead, the architecture requires a DOM Abstraction Layer, a structural wrapper that translates raw HTML nodes into standardized, semantic objects.3.1 Adaptive Queries and Virtual RepresentationThe DOM Abstraction Layer scans the document and constructs a virtual representation of all interactive elements. Semantic Field Detection and Accessibility ParsingTo accurately determine the purpose of a form field without relying on CSS selectors, the engine must simulate the interpretation mechanisms of an assistive technology device, such as a screen reader. This involves executing a prioritized parsing algorithm based on the W3C Accessible Name and Description Computation specification.4.1 The Accessible Naming HierarchyThe semantic field detection module evaluates a strict hierarchy of HTML attributes and relational nodes to extract the "Accessible Name" of any given input. This calculated name serves as the primary string utilized for subsequent data mapping.Evaluation PriorityMechanismArchitectural JustificationFirst Priorityaria-labelledbyThis attribute explicitly points to the ID of another element containing the label text. It allows a single text node to label multiple inputs, or multiple text nodes to be concatenated into a single label. It represents the highest degree of developer intentionality.Second Priorityaria-labelThis attribute provides a direct, hidden string value for the accessible name, overriding standard visible labels. It is critical for identifying fields where visual labels are intentionally omitted for design purposes, such as standalone search or submission icons.Third PriorityNative <label> ElementThe standard HTML <label> tag, either wrapping the input implicitly or explicitly associated via the for attribute matching the input's id. This remains the most common and robust native semantic association.Fourth Priorityplaceholder AttributeWhile the W3C strongly discourages using placeholders as primary labels due to accessibility concerns—specifically because the text vanishes upon user interaction and typically fails contrast requirements—it remains a highly prevalent practice in modern UI design and must be ingested as a fallback heuristic.Supplemental Contextaria-describedbyThis attribute provides secondary context, often pointing to instructional text or validation requirements (e.g., "Format: YYYY-MM-DD"). While not the primary name, appending this string to the extracted label significantly improves the accuracy of the downstream NLP mapping engine.4.2 Visual Proximity and Bounding Box HeuristicsIn poorly constructed enterprise applications, explicit programmatic associations between labels and inputs are frequently omitted. Text nodes simply exist in visual proximity to the input fields. To resolve these orphaned inputs, the semantic detection engine implements a geometric proximity algorithm.The proximity engine utilizes the getBoundingClientRect() API to map the exact spatial coordinates of the target input. It subsequently scans the DOM for independent text nodes that do not belong to other interactive elements. By calculating the Cartesian distance between the outer edges of the text node's bounding box and the input's bounding box, the algorithm infers a relationship.The algorithm establishes strict tolerance thresholds. A text node is designated as the semantic label if it resides within a narrow vertical band directly above the input, or within a tightly constrained horizontal band immediately preceding the left edge of the input (in left-to-right languages). To prevent false positives in dense data tables, the threshold is dynamically adjusted based on the average margin and padding values computed from the host page's overall stylesheet geometry.5. Dynamic Field Mapping EngineOnce the semantic detection layer extracts the intended label of a field (for example, "Gross Annual Earnings"), the system must map this real-world string to the extension's standardized internal schema (for example, taxpayer.financials.yearlyIncome). Because web developers utilize endless variations of terminology, exact string matching and regular expressions are fundamentally inadequate. The architecture employs a dual-layered, intelligent mapping system consisting of high-speed lexical fuzzy matching and deep semantic embeddings.5.1 Lexical Distance and Fuzzy MatchingFor the initial mapping pass, the engine relies on lightweight, in-memory fuzzy string matching algorithms. Libraries such as Fuse.js or microfuzz are optimally suited for the MV3 environment, as they execute synchronously, require zero external dependencies, and impose minimal memory overhead.These algorithms operate by calculating the edit distance between the detected DOM label and the predefined aliases within the internal schema. The Levenshtein distance quantifies the precise number of single-character operations—insertions, deletions, or substitutions—required to mutate one string into another. This is exceptionally effective at resolving typographical errors or minor stylistic variations, easily matching "Email Address" to "e-mail address".To further enhance accuracy, the engine heavily biases the Jaro-Winkler distance algorithm. The Jaro-Winkler calculation applies an increased mathematical weight to matching prefixes. In the context of form fields, prefixes are highly indicative of intent; a field labeled "Telephone (Mobile)" will score highly against the schema alias "Telephone Number" because the leading characters perfectly align, allowing the engine to successfully disregard the trailing parenthetical variations.5.2 On-Device Semantic AI EmbeddingsWhile lexical fuzzy matching resolves typographical differences, it is fundamentally incapable of understanding conceptual synonyms. A lexical algorithm will fail to connect the DOM label "Wages" to the schema key "Salary" because they share no character sequences. To achieve true human-like comprehension, the advanced phase of the architecture integrates a local Natural Language Processing (NLP) embedding model.Due to the strict privacy constraints prohibiting the transmission of sensitive taxpayer data to external backend APIs, the NLP layer must execute entirely on-device. This is accomplished using Transformers.js, a library that leverages WebAssembly (Wasm) and the WebGPU API to execute neural networks directly within the Chrome Extension sandbox.The system utilizes highly quantized, ultra-lightweight embedding models, such as the Xenova/all-MiniLM-L6-v2 or the EmbeddingGemma-300M, which are explicitly optimized for edge-device deployment. Because MV3 Content Security Policies prevent the dynamic fetching of executable Wasm code from remote Content Delivery Networks during runtime, the ONNX model files and their associated WebAssembly binaries must be statically bundled within the extension's local directory and explicitly declared in the web_accessible_resources directive of the manifest.The embedding model transforms both the extracted DOM label and the internal schema aliases into high-dimensional, dense floating-point vectors. The engine then calculates the Cosine Similarity between these vectors, assessing the angle between them in the multi-dimensional semantic space. If the vectors point in the same general direction, indicating high semantic equivalence, the resulting cosine similarity score will approach a value of 1.0. This allows the system to confidently map "Total Remuneration" on the host website to the internal income variable without any predefined programmatic link.5.3 Confidence Scoring and Mapping ResolutionTo manage ambiguity and prevent the catastrophic failure of submitting incorrect data to a government portal, both the lexical and semantic engines output a normalized confidence score between zero and one hundred percent.When multiple schema keys evaluate as potential matches for a single DOM field, the system initiates a resolution protocol. It first attempts to use the broader context of the form element—such as the <legend> text of the enclosing <fieldset>—to disambiguate the field. If a field labeled "Address" belongs to a fieldset labeled "Employer Details", the engine boosts the score for the employer.address schema key over the personal.address key.The final, aggregated confidence score dictates the system's operational behavior through a strict rubric:Confidence Score ThresholdEngine Action ProtocolUser Interface AugmentationHigh (85% - 100%)Automated ExecutionThe system programmatically fills the input. A shadow-DOM tooltip renders beside the element, displaying the inferred mapping and requiring explicit user confirmation before proceeding.Low (0% - 59%)Graceful DegradationThe automation routine suspends. The extension exposes a custom, searchable dropdown menu over the element, allowing the user to manually override the engine and select the correct data point from their profile.6. Mutation-Aware Automation and PerformanceTraditional web automation scripts frequently rely on static timing events, such as the DOMContentLoaded event, to trigger field scanning. However, modern Single Page Applications (SPAs) utilize asynchronous data fetching, meaning the initial HTML payload is often empty, and form fields are injected into the DOM by JavaScript long after the native load events have concluded. Hardcoding arbitrary setTimeout delays is universally recognized as an anti-pattern, as it either wastes time on fast connections or fails entirely on slow networks.6.1 The MutationObserver ParadigmTo achieve true resilience, the architecture employs the native MutationObserver API. This interface allows the extension to asynchronously monitor the DOM tree for the real-time addition, removal, or modification of elements. The MutationObserver provides absolute timing precision, reacting at the exact millisecond a new form step or dynamic field is rendered by the host application's framework.JavaScript// Robust Semantic Element Waiter utilizing MutationObserver
function waitForSemanticField(targetSemanticName, timeoutMs = 15000) {
    return new Promise((resolve, reject) => {
        // Immediate synchronous check
        const existingMatch = scanForField(targetSemanticName);
        if (existingMatch) return resolve(existingMatch);

        const observer = new MutationObserver((mutations) => {
            // Filter out text changes or attribute changes to save CPU cycles
            const hasStructuralAdditions = mutations.some(m => m.addedNodes.length > 0);
            if (!hasStructuralAdditions) return;

            // Debounce the heavy semantic parsing using requestAnimationFrame
            window.requestAnimationFrame(() => {
                const match = scanForField(targetSemanticName);
                if (match) {
                    observer.disconnect();
                    resolve(match);
                }
            });
        });

        // Limit observation scope to the body body to prevent memory leaks
        observer.observe(document.body, { childList: true, subtree: true });

        // Fallback rejection to prevent infinite hangs
        setTimeout(() => {
            observer.disconnect();
            reject(new Error(`Timeout elapsed waiting for field: ${targetSemanticName}`));
        }, timeoutMs);
    });
}
6.2 Performance Optimization StrategiesWhile MutationObserver is significantly faster than the deprecated Mutation Events, monitoring the entire document recursively (subtree: true) on a highly complex enterprise portal can lead to severe performance degradation if not meticulously managed. The callback function is executed as a microtask, meaning an unoptimized parsing algorithm could block the browser's main thread, causing the host page to freeze during rendering.To mitigate this, the engine implements strict performance optimizations. The callback specifically filters the MutationRecord array to ignore attribute modifications or character data changes unless explicitly required, focusing solely on childList additions. Furthermore, the engine entirely avoids executing layout-thrashing properties within the observer callback, such as accessing offsetTop or getBoundingClientRect(), which force the browser to execute synchronous layout recalculations. Instead, visual proximity calculations are deferred and batched using window.requestAnimationFrame, ensuring that heavy geometric math only occurs when the browser has idle main-thread capacity.7. Advanced Input Simulation for Modern FrameworksThe most frequent point of failure in naive form automation is the methodology used to insert data into an input field. Executing a simple assignment like document.getElementById('tax').value = '1000' modifies the physical HTML element, but it completely bypasses the internal state management mechanisms of modern JavaScript frameworks like React, Vue, or Angular.In a React application, for example, the framework maintains an internal Virtual DOM and deliberately overrides the native setter functions of HTML inputs to enforce one-way data binding. If an extension forcefully changes the HTML value without notifying the framework, the framework's internal state remains null. When the user attempts to submit the form, the application evaluates its internal state, determines the fields are empty, and triggers false validation errors.7.1 Bypassing Synthetic Event HijackingTo resolve this, the automation engine must forcibly synchronize the physical DOM with the host framework's internal memory. This requires extracting the native property descriptor from the element's prototype chain, invoking the unadulterated setter method, and subsequently dispatching a sequence of native bubbling events to trick the framework's synthetic event listeners into recording the update.JavaScript// Engine Utility to simulate human typing in a framework-agnostic manner
function simulateFrameworkInput(element, value) {
    // Trigger any onFocus validation overlays or tooltips
    element.focus();

    // Extract the native prototype setter, bypassing React/Angular overrides
    const nativeValueSetter = Object.getOwnPropertyDescriptor(
        Object.getPrototypeOf(element), 
        'value'
    ).set;
    
    // Inject the value natively
    nativeValueSetter.call(element, value);

    // Dispatch the input event (Primarily used by React and Vue)
    element.dispatchEvent(new Event('input', { bubbles: true, cancelable: true }));

    // Dispatch the change event (Primarily used by Angular and vanilla JS)
    element.dispatchEvent(new Event('change', { bubbles: true, cancelable: true }));
    
    // Trigger onBlur to satisfy "touched" validation requirements
    element.blur();
}
7.2 Simulating Complex Interactive WidgetsEnterprise portals frequently deploy custom UI widget libraries, such as KendoUI or DevExpress, for complex inputs like multi-select dropdowns or interactive date pickers. These components typically hide the actual HTML <input> element and render a stylized interface using nested <div>, <span>, and <svg> elements.For these intricate components, dispatching an input event is insufficient. The engine must simulate the exact sequence of low-level mouse interactions a human would perform to trigger the dropdown overlay's visibility. This requires dispatching a precise chronological sequence of MouseEvent objects: mousedown, followed immediately by mouseup, and concluding with a click event, all injected at the exact center coordinates of the target element's bounding box.Once the customized dropdown menu is forced into the DOM, the MutationObserver detects the newly injected list items, the semantic engine identifies the correct option text, and the engine executes a secondary simulated mouse sequence on the target option, perfectly emulating human interaction.8. Multi-Step Form State ManagementTax applications generally deploy lengthy, paginated wizards to compartmentalize data entry into logical sections (e.g., Personal Details, Income Declarations, Deductions). Successfully automating these flows requires the extension to maintain persistent situational awareness across both soft SPA transitions (where the URL changes via the History API without a page reload) and hard navigational redirects.8.1 Finite State Machine OrchestrationThe system architecturally models the multi-step form as a Finite State Machine (FSM) coordinated by the background Service Worker. Upon the initiation of an autofill sequence, the Service Worker serializes the target user profile and an "Execution Plan" into the chrome.storage.session API. The Execution Plan defines the ordered sequence of expected logical steps.When the Content Script initializes on a new page or detects a massive DOM replacement indicating a soft navigation, it queries the Service Worker for the active state payload. The Content Script then evaluates the current DOM against a set of heuristic signatures—such as scanning for specific <h1> headers or unique field clusters—to determine the current active step within the Execution Plan.Upon successfully populating all mapped fields for the active step, the Content Script updates the progression index within the session storage and invokes the simulated click routine on the recognized "Next" or "Continue" button. Because the state is securely maintained in MV3 storage, if the portal enforces a hard page reload, the freshly injected Content Script seamlessly retrieves the updated index and resumes execution precisely where the previous script terminated.8.2 Intelligent Retry MechanismsNetwork latency, backend server processing delays, or unexpected asynchronous validation errors can easily cause a step transition to fail. A robust engine does not rely on fatal exceptions that crash the script. Instead, the automation engine implements an exponential backoff retry loop.Following a transition click, the engine monitors the DOM for the appearance of the next step's signature elements. Concurrently, it actively scans for the injection of error indicators, utilizing accessibility heuristics such as the sudden appearance of aria-invalid="true" attributes on previously filled fields, or the rendering of elements tagged with role="alert". If a validation error is intercepted, the FSM halts the autonomous progression sequence, shifts the extension into the Graceful Degradation mode, and flags the problematic field for manual human correction.9. Debugging, Logging, and Visual TelemetryOperating a complex, asynchronous automation engine inside the MV3 isolated world presents severe diagnostic challenges. Background Service Workers cannot natively render console.log statements to the user's active browser console, rendering traditional debugging methods ineffective. A resilient architecture mandates a dedicated, in-extension telemetry and logging framework.9.1 Event Sourcing and Serialized LogsThe system implements an internal event sourcing methodology. Every significant action executed by the engine—including raw DOM extractions, fuzzy matching similarity scores, Wasm embedding generation times, exact payload insertions, and MutationObserver timeout triggers—is serialized into a structured JSON event object. These chronological logs are batched and flushed to chrome.storage.local. A dedicated debugging panel within the extension's Options page permits developers to extract and download these logs, facilitating the rapid post-mortem diagnosis of execution failures on complex, authenticated government domains without requiring direct engineering access to the user's account.9.2 Shadow DOM Visual Debug OverlaysFor end-users and quality assurance personnel, the engine provides a real-time Visual Debug Mode. When activated, the Content Script injects an isolated Shadow DOM tree over the host page. The Shadow DOM boundary guarantees that the heavy CSS styling required for the debugging overlays does not leak into the host page and inadvertently corrupt the portal's layout.As the semantic engine traverses the interactive elements, it projects semi-transparent, color-coded bounding boxes directly over the target inputs. These visual overlays dynamically display the engine's real-time calculations. A user can hover over a yellow-bordered field to view a Shadow DOM tooltip detailing exactly why the engine hesitated—displaying the extracted accessible name, the closest internal schema alias, and the specific decimal confidence score generated by the cosine similarity calculation. This transparent telemetry builds user trust and drastically reduces the time required to maintain the mapping schemas as the government portal evolves.10. Implementation Roadmap and System EvolutionTo effectively deploy this system within a production environment, the engineering rollout should follow a phased architectural roadmap, separating the core infrastructure from the advanced heuristic capabilities.Phase 1: Minimum Viable Product (MVP) PlatformThe MVP focuses entirely on establishing a compliant, robust MV3 foundation. It implements the Service Worker for asynchronous state coordination and utilizes chrome.storage.session for rapid data hydration. The DOM abstraction layer relies on standard Accessible Naming calculations, omitting the complex geometric proximity algorithms. Field mapping is restricted to high-speed, purely lexical fuzzy matching using Fuse.js. Finally, the input simulation utilizes the prototype bypass pattern to ensure fundamental form submissions succeed on the React or Angular-driven RAMIS portal, bypassing the most common synthetic event traps.Phase 2: Advanced System AI and Resilience IntegrationThe subsequent phase transitions the MVP into a highly intelligent, self-healing automation engine. The primary addition is the integration of Transformers.js and lightweight quantized ONNX models via WebAssembly. This introduces semantic embedding mapping, allowing the engine to correctly map entirely unrecognized fields based on contextual meaning rather than character strings. The MutationObserver logic is strictly optimized with requestAnimationFrame debouncing to ensure zero impact on host page performance. The spatial bounding-box algorithms are deployed to parse legacy layouts lacking explicit ARIA labels. Finally, the Visual Debug Mode is fully integrated into the Shadow DOM, providing comprehensive, real-time telemetry overlays to the user, ensuring the engine degrades gracefully into a hybrid human-machine workflow when encountering absolute ambiguity.12. ConclusionThe development of a highly reliable form automation extension for volatile, dynamic government portals fundamentally requires abandoning traditional, developer-centric DOM manipulation tactics. The strict architectural boundaries imposed by Chrome Manifest V3—specifically the ephemeral nature of Service Workers and the prohibition of remote code execution—necessitate a paradigm shift toward stateless orchestration and completely on-device intelligence.By meticulously combining W3C semantic accessibility evaluations, in-browser NLP embedding models for conceptual field mapping, advanced prototype manipulation to bypass frontend framework event hijacking, and highly optimized, mutation-aware execution loops, the proposed architecture establishes a system that perceives and interacts with web forms much like a human operator. This engine transforms brittle, high-maintenance automation scripts into an adaptive, self-healing ecosystem capable of navigating complex, multi-step Single Page Applications efficiently, accurately, and entirely within the secure confines of the user's browser.
